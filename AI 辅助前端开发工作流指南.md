## 1. 头脑风暴 — 理解产品需求（PRD）

### 目标

将自然语言形式的产品需求转化为结构清晰、无歧义的开发任务。

### 操作步骤

1. **提供 PRD 内容**：将需求文档的核心章节（功能描述、用户故事、验收标准）复制到 AI 对话，或截图拖入。

2. **让 AI 提炼关键信息**：

   ```
   以下是产品 PRD，请帮我：
   1. 用一句话概括核心目标
   2. 列出所有新增 / 修改的页面或功能点
   3. 列出明确的验收标准（AC）
   4. 标出文档中模糊或缺失的信息，我来补充
   ```

3. **逐条澄清边界**：AI 针对不明确的地方主动提问，覆盖异常流程、权限控制、多端差异等边界条件。

4. **输出结构化变更提案**：确认无歧义后，将需求固化为一份结构化文档（任务列表 + 验收标准），作为后续所有阶段的唯一事实来源。

### 产出物

- 功能边界说明
- 任务拆解清单
- 验收标准（AC）列表

## 2. Plan — 消化 UI 设计稿

### 工具

- **Figma MCP**：直接读取 Figma 设计文件，提取布局、样式、组件结构。

### 操作步骤

1. **获取 Figma 节点链接**：从设计同事处拿到对应页面或组件的 Figma 链接。

2. **调用 Figma MCP 读取设计**：在 AI 对话中粘贴链接，AI 自动拉取设计内容，返回：
   - 组件截图
   - 布局结构（尺寸、间距、颜色 Token）
   - 已接入 Code Connect 的组件映射关系

3. **适配项目技术栈**：告知 AI 当前项目使用的 UI 框架和组件库，让其将设计参考转化为符合项目规范的代码：

   ```
   请根据上方 Figma 设计，用 [项目 UI 框架] 实现这个页面。
   优先复用项目已有的公共组件，遵循项目代码规范。
   ```

4. **确认交互细节**：对于 hover、loading、空数据、错误态等，逐一确认是否有设计稿；没有的按项目惯例处理。

### 产出物

- 组件结构方案
- 样式实现参考
- 交互状态清单

## 3. Work Plan — 解析后端接口文档

### 工具

- **WebFetch / 文档读取**：AI 直接抓取在线 Swagger / OpenAPI 文档。

### 操作步骤

1. **提供接口文档地址**：

   ```
   请帮我分析以下接口文档，提取与「[功能名称]」相关的接口：
   https://<env>.example.com/swagger-ui/index.html
   ```

   若文档需要登录，先导出 JSON / YAML 文件后粘贴给 AI。

2. **AI 提取并整理接口信息**：

   | 字段           | 说明                      |
   | -------------- | ------------------------- |
   | Method         | GET / POST / PUT / DELETE |
   | Path           | `/api/v1/...`             |
   | Request Params | 字段名、类型、是否必填    |
   | Response       | 核心字段结构              |
   | 错误码         | 业务异常码及含义          |

3. **生成接口调用层骨架**：AI 根据整理好的接口信息，自动生成对应的服务函数和 TypeScript 类型定义。

4. **拆解开发任务**：结合 PRD 产出的任务清单和接口文档，让 AI 将功能点与接口一一对应，形成最终的 Work Plan。

### 产出物

- 接口字段说明表
- 接口调用层代码骨架
- 请求 / 响应 TypeScript 类型
- 开发任务与接口的对应关系

## 4. 编码 — 多语言处理（i18n MCP）

### 背景

内部 i18n MCP Server 提供两个核心能力：
1. **创建翻译 Ticket**：为新增文案在翻译平台上创建任务
2. **上传到服务端**：将文案及 i18n key 推送至翻译管理系统

整个流程由 AI 全程自动化处理，无需人工介入。

### 操作步骤

1. **编码时使用 i18n Key**：所有面向用户展示的文案均通过 i18n 函数调用，禁止直接写字面量字符串。

2. **开发完成后，让 AI 扫描新增文案**：

   ```
   请扫描本次改动涉及的文件，列出所有新增的 i18n key 及其默认文案（表格形式）。
   ```

3. **AI 自动调用 i18n MCP 完成全流程**：

   ```
   请调用 i18n MCP，对上方列出的所有新增文案：
   1. 在翻译平台创建对应的翻译 Ticket
   2. 将文案和 i18n key 上传到服务端
   ```

   AI 将依次完成：创建 Ticket → 上传文案 → 返回每条 key 的处理结果。

4. **确认处理结果**：AI 输出每条 key 的上传状态，失败项自动重试或给出错误原因。

### 注意事项

- 上传前确认 key 命名规范（如：`模块名.组件名.描述`）。
- 先搜索是否已存在相同文案的 key，避免重复创建。

### 产出物

- 新增 i18n key 清单
- 翻译 Ticket 链接列表
- 服务端上传结果报告

## 5. Test — 浏览器 MCP 自动化测试

### 工具

- **Chrome DevTools MCP**：通过 MCP 协议控制浏览器，执行端对端自动化测试。

### 操作步骤

1. **启动本地或测试环境**：确保应用已运行并可访问。

2. **描述测试场景，让 AI 执行**：

   ```
   请打开 http://localhost:3000/[页面路径]，按以下步骤测试「[功能名称]」：
   1. [操作步骤一]，确认 [预期结果]
   2. [操作步骤二]，确认 [预期结果]
   3. 测试 [异常场景]，确认 [错误处理]
   ```

3. **AI 执行并截图反馈**：Chrome MCP 在每个关键步骤截图，AI 对比预期与实际状态，标出通过 / 失败。

4. **覆盖边界场景**：

   | 场景类型 | 验证点                      |
   | -------- | --------------------------- |
   | 网络异常 | 是否有 loading 态和错误提示 |
   | 权限不足 | 是否正确隐藏或禁用操作      |
   | 空数据态 | 是否展示空状态引导          |
   | 输入校验 | 是否有完整的表单校验反馈    |
   | 大数据量 | 是否有分页 / 虚拟滚动       |

5. **记录失败项**：测试失败的用例整理为问题清单，供修复参考。

### 产出物

- 测试步骤截图记录
- 通过 / 失败用例报告
- 待修复问题清单

## 6. Review — 多 Agent 并行 Code Review

### 策略

使用 **Best-of-N** 模式：同时启动多个独立 Agent，各自从不同视角审查同一份代码，最终汇总结论，比单一 Agent 审查覆盖面更广、发现问题更全面。

### 操作步骤

1. **准备 Review 上下文**：导出本次改动的 diff：

   ```bash
   git diff main...HEAD
   ```

2. **并行启动多个 Agent**，分配不同职责：

   | Agent              | 职责                            | Prompt 关键词                                               |
   | ------------------ | ------------------------------- | ----------------------------------------------------------- |
   | **安全 Agent**     | XSS、敏感信息泄露、权限校验缺失 | "审查安全漏洞，关注输入过滤、鉴权逻辑、敏感数据暴露"        |
   | **性能 Agent**     | 无效渲染、内存泄漏、请求优化    | "审查性能问题，关注重复渲染、无效依赖、大数据处理"          |
   | **规范 Agent**     | 命名、注释、代码结构、团队约定  | "审查代码规范，关注命名一致性、重复代码、文件组织"          |
   | **业务逻辑 Agent** | 实现是否与需求 / 设计完全一致   | "对照以下验收标准，审查实现是否完整覆盖所有场景：[粘贴 AC]" |

3. **汇总 Review 结论**：所有 Agent 完成后，让主 Agent 汇总问题并按优先级分类：
   - �� **必须修复**（Blocker）：影响功能正确性或安全性
   - �� **建议优化**（Non-blocker）：影响可维护性或性能
   - �� **记录存档**（Info）：技术债或后续改进点

4. **逐条修复**：针对 Blocker 问题，逐一给出修复方案并执行；修复后对对应维度重新 Review 确认。

### Agent Prompt 模板

```
你是一名资深工程师，负责对以下代码进行 [安全 / 性能 / 规范 / 业务逻辑] 维度的 Code Review。

请审查以下 diff，输出结构化问题清单，每条包含：
- 问题描述
- 所在文件和行号
- 严重程度（Blocker / Non-blocker / Info）
- 修复建议

[粘贴 git diff 内容]
```

### 产出物

- 分维度 Review 问题清单
- 分级汇总报告（Blocker / Non-blocker / Info）

## 7. PR — 提交合并请求

### 操作步骤

1. **最终自检**：确保本地测试通过、lint 无报错、类型检查通过。

2. **让 AI 生成 PR 描述**：

   ```
   请根据以下 git log 和 diff，生成一份标准 PR 描述，包含：
   - 变更背景（关联需求）
   - 改动范围
   - 测试方法
   - 截图（如有 UI 改动请说明截图位置）
   ```

3. **提交 PR**：

   ```bash
   gh pr create --title "feat: [模块] 功能描述" --body "$(cat pr-body.md)"
   ```

4. **关联需求文档**：在 PR 描述中附上第 1 阶段产出的结构化变更提案链接，确保需求到代码的完整可追溯性。

### 产出物

- Pull Request（含完整描述）
- 变更可追溯性记录
